Technical Architecture & Planning Document — Next.js Demo UI + LLM-Native Copy Assistant
Date: 4 March 2026 Version: 1.0 Status: Pre-Implementation Review

1. Executive Summary
This document describes the architecture for a Next.js Demo UI + LLM-Native Copy Assistant — a lightweight, full-stack application that demonstrates a modern pattern for AI-driven personalisation:

"This visitor just opened this page. Can we generate a highly relevant headline using an LLM, but strictly constrain it to only state facts we know are true?"

Instead of letting an LLM generate unbounded, potentially hallucinated text, this system provides the LLM with a strict list of allowed "claims" (facts or features) and forces it to cite which claims it used. A strict validation boundary guarantees the LLM's output is safe to display. This is the foundation for safe, enterprise-grade AI content generation.

2. The Big Picture — How Constrained LLM Generation Works
2.1 The Runtime Flow
When a user opens the landing page, the following happens in real time:

User opens the website ↓
Frontend sends visitor context & desired rules to the API ↓
API receives request at POST /decide (implemented via Next.js API route) ↓
API loads allowedClaims and prompts the LLM ↓
LLM returns JSON (headlines + usedClaimIds) ↓
API validates that usedClaimIds ⊆ allowedClaims ↓
If valid, API returns headlines; if invalid, it retries or falls back ↓
Frontend renders the safe variant to the user
2.2 Why Use This Pattern
Industry Use Case	What It Does
Brand Safety	Prevents LLMs from inventing features or pricing that do not exist.
Legal Compliance	Ensures financial or health claims generated by AI are pre-approved by legal teams.
Personalisation	Allows infinite copy variations tailored to the user, without infinite risk.
3. Key Concepts
3.1 What Is an Allowed Claim?
An allowed claim is a pre-approved piece of information that the LLM is permitted to use when generating copy.

Claim ID	Text
claim-speed	"Our product is 10x faster than competitors."
claim-price	"Starts at $9/month."
claim-security	"SOC2 Type II Certified."
3.2 What Is the Model Output?
The LLM does not return raw Markdown or plaintext. It is explicitly prompted to return a strictly structured JSON payload matching the following schema:

{
  "headline1": "Experience 10x faster performance today.",
  "headline2": "Secure your data with SOC2 compliance.",
  "usedClaimIds": ["claim-speed", "claim-security"]
}
3.3 What Is the Validation Boundary?
The core architectural constraint. Before the backend sends the generated copy to the frontend, it verifies the LLMs "work". If the LLM returns usedClaimIds: ["claim-speed", "claim-fake"], the system detects claim-fake is not in the allowed list, and instantly rejects the payload.

4.2 The Two API Endpoints
The application uses a unified Next.js App Router setup.

Endpoint: POST /decide (implemented via Next.js API route /api/decide) The frontend sends the visitor's context and any manual demo rules. The Admin page provides a preview mechanism for 2–3 preset visitors.

Example Preset Visitors:

Mobile Egypt: country=EG deviceType=mobile
Google Visitor: referrerDomain=google.com
Default (Fallback): no rules match
Example Demo Rules:

tone: Professional, Playful
length: Short, Detailed
emphasis: Urgency, Trust
The endpoint performs three steps:

Fetch: Gather the allowedClaims.
Generate: Call the LLM with the constraints, context, and claims.
Validate: Run the validation boundary check.
5. The Validation Boundary — Safety by Design
5.1 Why the Boundary Matters
LLMs are probabilistic. They will occasionally ignore instructions, hallucinate new features, or output invalid JSON. The boundary treats the LLM as an untrusted input source.

5.2 The Subset Rule
The mathematical rule governing this architecture is: usedClaimIds ⊆ allowedClaims

If usedClaimIds contains any string not perfectly matching an ID in allowedClaims, the entire generated payload is considered toxic and is rejected.

5.3 Practical Consequence
If the marketing team wants the AI to stop mentioning a specific discount, they simply remove claim-discount from the allowedClaims array. The validation boundary guarantees the LLM cannot safely output text using that claim anymore, providing absolute control over AI output.

6. Resilience Strategy (Retry & Fallback)
What happens when the LLM fails the boundary check? A single failure should not break the UI.

Attempt 1: Call LLM -> Validate. If success, return. If fail, proceed to Attempt 2.
Attempt 2 (Retry): Re-call LLM (optionally appending an error message instructing it to fix its mistake) -> Validate. If success, return. If fail, proceed to Fallback.
Fallback: Return perfectly safe, hardcoded default text (e.g., "headline1": "Welcome to our platform.", completely bypassing the LLM).

2. Prompt	LLM generates copy & JSON	Outputs usedClaimIds: ["claim-security"]

Step	What Happens	Data
1. Request	Frontend sends context	visitor: "Startup"
2. Prompt (attempt 1)	LLM generates copy & JSON	Outputs usedClaimIds: ["claim-speed", "claim-ai"]
3. Validation 1	Engine checks subset	claim-ai is missing. -> FAIL
4. Prompt (attempt 2)	System automatically retries	Outputs usedClaimIds: ["claim-speed"]
5. Validation 2	Engine checks subset	["claim-speed"] ⊆ [...] -> PASS
6. Response	API returns payload	headline1: "Build 10x faster."
8. API Contracts
POST /decide
Purpose Given a visitor profile and rule tweaks, generate safe LLM copy.

Request Body

Field	Type	Required	Description
visitorProfile	string	Yes	E.g., "New Visitor", "Returning User"
rules.tone	string	No	Demo control: "Professional", "Playful"
rules.length	string	No	Demo control: "Short", "Detailed"
rules.emphasis	string	No	Demo control: "Urgency", "Trust"
Response Body (Success / Validated / Fallback)

Field	Type	Description
headline1	string	The primary generated headline
headline2	string	A secondary generated headline/subheading
usedClaimIds	string[]	The claim IDs the LLM utilized
isFallback	boolean	true if generation failed twice and defaults were used
8.1 Logging and Safety (README Requirement)
The system satisfies the logging requirement by detailing the following in the README.md:

Logging and Safety The application logs all LLM interactions and validation outcomes to the server console. Each generation includes a trace containing:

the allowed claims provided to the model
the raw model output
the validation result
If the validation boundary detects that usedClaimIds contains values not present in allowedClaims, the payload is rejected and the system automatically retries once. If the retry fails, the system falls back to a safe default response.

12. Testing Strategy
12.1 Manual Verification Points
Start the server and open http://localhost:3000. Verify the default visitor profile triggers a successful LLM call and text rendering.
Open the Admin preview panel /admin. Change Tone to "Playful" and Visitor to "Returning". Click "Preview". Verify the generated text reflects the changes.
Simulate Validation Failure: Temporarily hardcode the LLM to return ["fake-claim"] in lib/llm.ts. Watch the server logs. Prove that the system logs Validation Failed, attempts Retry 1, fails again, and returns the expected Fallback JSON payload to the frontend without crashing.
